retrofit

retrofit是square的又一个强大的库，他的特点在于利用接口，方法，注解参数来声明式定义一个请求应该如何被创建，这样的组织形式使得在本地编写api的时候更加清晰。

Rest

REST(Representational State Transfer)是一种设计思路，它首次出现在2000年Roy Fielding的博士论文中，Roy Fielding是 HTTP 规范的主要编写者之一。 他在论文中提到:“我这篇文章的写作目的，就是想在符合架构原理的前提下，理解和评估以网络为基础的应用软件的架构设计，得到一个功能强、性能好、适宜通信的架构。REST指的是一组架构约束条件和原则。”

REST本身并没有创造新的东西，而是对旧有的资源进行整合。目前来说只适用于web，HTTP是目前唯一一个与REST绑定的实例，有三个重要的概念:资源,表现层,状态转化。

资源

任何我们所需要的事物都可以被称为资源，无论格式，内容。URI（Uniform Resource Identifier）在Web中会对资源进行唯一标识，根据URI我们便可以获取到资源。 URI的设计应该遵循可寻址性原则，具有自描述性，需要在形式上给人以直觉上的关联。 有一些技巧

  • 使用 _ 或 - 对URI中的单词进行分割，使得表达更加易懂
  • 使用／来表示层级关系
  • 使用？添加参数来进行资源的过滤

表现层

即资源展示形式，表示需要以什么格式来组织数据进行传输。在HTTP协议中通过报文头部的请求行“accept”和“content－Type”来设定

状态转化

访问一个网站，即了客户端和服务器的一个互动过程。在这个过程中，肯定涉及到数据和状态的变化。而HTTP协议是无状态的，如果客户端想要通知服务器端改变数据和状态的变化，肯定要通过某种方式来通知它。

HTTP中提供了8种方法常用的有四种，GET，POST，PUT，DELETE。简单来理解可以用数据库的操做来进行对应。

下面主要说明retrofit的注解，这个是其特色，整体采用了rest设计，所以在前面提前说了下rest。

注释

方法注释

  • 方法类 @GET，@POST，@PUT，@DELETE 后面加URL
  • 方法注释 @Headers 里面直接添加请求行
  • 方法注释 @Http 其拥有三个属性：method，path，hasBody

    请求头注视

  • 参数注释 @Header value 为请求行名称，该注解会更新已有的请求头
  • 方法注释 @Headers value 为多个请求头，这些请求头固定

    请求和响应格式注解

  • 方法注释 @FormUrlEncoded 表示请求发送编码表单数据，每个键值对需要使用@Field注解
  • 方法注释 @Multipart 表示请求发送multipart数据，需要配合使用@Part
  • 方法注释

    请求参数类注解

  • 参数注释 @Query @QueryMap 用于Http Get请求传递参数.
  • 参数注释 @Field @FiledMap 用于Post方式传递参数,需要在请求接口方法上添加@FormUrlEncoded,即以表单的方式传递参数
  • 参数注释 @Part @PartMap 用于表单字段,Part和PartMap与Multipart注解结合使用,适合文件上传的情况
  • 参数注释 @Body 用于Post,根据转换方式将实例对象转化为对应字符串传递参数.
  • 参数注释 @Path 用于URL中替位符 @GET(“group/{id}/users”) ｛｝中即为替换值
  • 参数注释 @URL 根据标记的URL进行请求

RxJava

RxJava的应用场景很多,在扔物线和dddd864140130的文章中提到了几种,包括RxBinding,RxLifecycle,RxBus,retrofit等等
RxBinding封装了原先android的时间体系,RxLifecycle是封装了生命周期,RxBus封装了EventBus,在项目中我更多使用到的是Rxjava对data的分装，和retrofit的绑定使用，特别是在后台自己做的时候。

  • 线程切换,利用Schedulers来切换线程，可以很轻松的在不同线程间切换，逻辑变得清晰，代码变得简洁
  • map 对网络返回的Respone映射
  • flatmap 目前大部分服务端接口设计都是通过用户名和密码登录获取access token，后面其他api的请求都是借助该token。对于需要注册功能的产品来说，我们经常面对这样的问题：使用用户名和密码登录成功后，保存服务器返回的access token，再调用服务端接口获取用户的详情信息
  • merge 很多情况下，一个界面中需要的数据来自多个数据源（请求），而只有当所有的请求的响应数据都拿到之后才能渲染界面。
  • concat 构建多级缓存，利用takeFirst，和merge的区别参看ReactiveX
  • timer 定时任务
  • interval 周期性任务
  • filter 过滤
  • 内存泄漏,当我们不需要的时候，主动取消订阅，原因在于Observable内部持有subscriber引用，而在subscriber中可能会持有view之类的引用
  • 封装第三方网络api，利用create方法
