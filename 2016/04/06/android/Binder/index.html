Binder

Binder是android系统最重要的特性之一。学习IPC时，必然要接触到android中这种IPC的主要方式。说Binder重要，是因为Android系统的四大组件，AMS，PMS等系统服务无一不与Binder挂钩。

理解了Binder对于理解整个android系统有着很大的帮助

binder在android中代表很多意思，直观上，Binder是android的一个类，实现了IBinder接口。从IPC角度来说，Binder是android的一种跨进程通信方式。Binder也可以理解为一种虚拟设备，有着驱动/dev/binder。从Android Framework角度来说，Binder是ServiceManager连接各种Manager（ActivityManager、WindowManager）和相应Service的桥梁。从Android应用层来说，Binder是客户端和服务端进行通信的媒介，可以通过其获取服务和数据。

Linux相关知识

android系统是基于linux内核的，我们首先需要了解一下系统底层的特性才好理解Binder。

进程隔离

进程隔离是为保护操作系统中进程互不干扰而设计的一组不同硬件和软件的技术。这个技术是为了避免进程A写入进程B的情况发生。进程的隔离实现，使用了虚拟地址空间。进程A的虚拟地址和进程B的虚拟地址不同，这样就防止进程A将数据信息写入进程B。

以上来自维基百科；操作系统的不同进程之间，数据不共享；因此一个进程需要与另外一个进程通信，需要某种系统机制才能完成。

用户空间/内核空间

Linux Kernel是操作系统的核心，可以访问受保护的内存空间，也有访问底层硬件设备的所有权限。

在系统中会对linux kernel进行保护，防止别的应用程序随意访问。这样就将Kernel和上层的应用分离开来，分别称为Kernel Space 和 User Space

系统调用/内核态/用户态

虽然系统对于内核有保护，然而不可避免的的是，总有那么一些用户空间需要访问内核的资源；比如应用程序访问文件，网络是很常见的事情

Kernel space can be accessed by user processes only through the use of system calls.

用户空间只能通过系统调用来访问内核空间，也就是所有的资源访问均在内核的控制下进行

当一个任务执行系统调用而陷入内核代码中执行的时候，称为内核运行状态（简称内核态）此时处理器处于特权级最高的（0级）内核代码中执行。当进程在执行用户自己的代码时，则称其处于用户运行态（用户态）。即此时处理器在特权级最低的（3级）用户代码中运行。处理器在特权等级高的时候才能执行那些特权CPU指令。

内核模块

传统的Linux通信机制内核支持，通过内核来实现不同用户空间之间的通信，Binder采取的是另一种方式，Linux有动态可加载内核模块（Loadable Kernel Module，LKM）机制；模块是具有独立功能的程序，它可以被单独编译，但不能独立运行。它在运行时被链接到内核作为内核的一部分在内核空间运行。这样，Android系统可以通过添加一个内核模块运行在内核空间，用户进程之间的通过这个模块作为桥梁，就可以完成通信了。

为何使用Binder

  1. 移动设备上大量CS通信方式的存在，原先linux下的IPC并不适用，只有socket是CS方式，然而socket主要用于网络和本机中进程的低速通信，效率低。
  2. 传输性能：linux上的IPC采用存储-转发方式，即数据先从发送方缓存区拷贝到内核开辟的缓存区中，然后再从内核缓存区拷贝到接收方缓存区，至少有两次拷贝过程。共享内存虽然无需拷贝，但控制复杂，难以使用。而采用Binder机制的话，则只需要进行1次拷贝即可，因为从发送方的缓存区拷贝到内核的缓存区，而接收方的缓存区与内核的缓存区是映射到同一块物理地址的，因此只需要1次拷贝即可
  3. 安全性：传统的Linux IPC机制没有任何安全措施，完全依赖上层协议来确保。传统IPC的接收方无法获得对方进程可靠的UID/PID，从而无法鉴别对方身份。而Binder机制则为每个进程分配了UID/PID来作为鉴别身份的标示，并且在Binder通信时会根据UID/PID进行有效性检测。

Binder通信模型

binder框架中定义了四个角色Client，Server，ServiceManager和Binder驱动。 这四个角色的关系和互联网类似：Server是服务器，Client是客户终端，SMgr是域名服务器（DNS），驱动是路由器。

  • binder驱动

    同路由器在网络中的地位相同，虽然默默无闻，但是做的确实最为核心的工作，缺少了Binder驱动通信就无法进行

  • ServiceManager

    ServiceManager和DNS的作用类似，里面存储着一张查找表，该表是server创建Binder后通知ServiceManager创建的，有了这个表Client就可以通过ServiceManager找到对应的server进行通信了。ServiceManager处于守护进程。那么问题来了，ServiceManager和server通信时用的也是跨进程通信，用Binder机制的话就有了先有鸡还是先有蛋的悖论。这个下面说

通信步骤

  1. 首先有一个进程向驱动提出申请为SM；驱动同意之后，SM进程负责管理Service
  2. XXXServer在自己的进程中向Binder驱动申请创建Binder实体。根据android中的使用来看应该是本地创建后通知驱动。
  3. Binder驱动为这个xxxServer创建内核中的Binder节点和Binder引用，然后将名字和引用打包传递给ServiceManager。通知SM注册一个名叫XXX的Service
  4. SM接到包取出名字和引用放入一张查找表中

  1. Client通过Service的名字在SM的查找表中获得Binder的引用了 Client也利用保留的handle值为0的引用向SM请求访问某个Service, SM就会从请求数据包中获得XXXService的名字，在查找表中找到该名字对应的条目，取出Binder的引用打包回复给client。

来说说上面的问题，Binder在处理这个问题时，采用了巧妙的方法。去创创造一置鸡来孵蛋：即在SM中事先创建好一个Binder实体，它的引用为一个固定值，所有人都知道。SM的特殊性在于，它与别的进程进行通信时，SM是server，别的是client，需要SM中Binder的引用来实现注册，查询和获取。当某个进程注册为SM时，Binder驱动会为其创建Binder实体，引用号都为0。就像DNS的地址，必须预先手工或动态配置好

Binder跨进程原理

Linux中大部分IPC（如AB通信）将A的数据Copy到内核中，B再从内核中获取

然而Binder的路不是这样的

假设Client进程想要调用XXXServer进程的object对象的一个方法add

注册流程就不说了，重点在Client查询。Client向SM查询：需要联系一个名为XXXService进程中的Object对象，进程之间通信的数据都会经过运行在内核空间里面的驱动，驱动在数据流过的时候做了一点手脚，它并不会给Client进程返回一个真正的object对象，而是返回一个看起来跟object一模一样的代理对象objectProxy，这个objectProxy也有一个add方法，但是这个add方法没有Server进程里面object对象的add方法那个能力；objectProxy的add只是一个傀儡，它唯一做的事情就是把参数包装然后交给驱动。

驱动收到这个消息，发现是这个objectProxy；一查表就明白了：我之前用objectProxy替换了object发送给Client了，它真正应该要访问的是object对象的add方法；于是Binder驱动通知Server进程，调用你的object对象的add方法，然后把结果发给我，Sever进程收到这个消息，照做之后将结果返回驱动，驱动然后把结果返回给Client进程；于是整个过程就完成了。

由于驱动返回的objectProxy与Server进程里面原始的object是如此相似，给人感觉好像是直接把Server进程里面的对象object传递到了Client进程；因此，我们可以说Binder对象是可以进行跨进程传递的对象

但事实上我们知道，Binder跨进程传输并不是真的把一个对象传输到了另外一个进程；传输过程好像是Binder跨进程穿越的时候，它在一个进程留下了一个真身，在另外一个进程幻化出一个影子（这个影子可以很多个）；Client进程的操作其实是对于影子的操作，影子利用Binder驱动最终让真身完成操作。

驱动中的Binder

  • binder_proc是描述进程上下文信息的，每一个用户空间的进程都对应一个binder_proc结构体
  • binder_node是Binder实体对应的结构体，它是Server在Binder驱动中的体现
  • binder_ref是Binder引用对应的结构体，它是Client在Binder驱动中的体现

他们的结构都是红黑树

有的地方把Binder本地对象直接称作Binder实体，把Binder代理对象直接称作Binder引用（句柄），其实指的是Binder对象在驱动里面的表现形式

Proxy

Proxy类则是通过一个IBinder类型的mRemote对象来跟驱动进行交互，并将对应的数据信息通过驱动与服务端的进程进行交互

构造方法和asBInder就不说了，主要说下接口方法的实现

proxy中实现的方法运行在客户端，当客户端调用时，内部的实现是：先创建方法需要的输入型Parcel对象data，输出型Parcel对象reply，返回对象（void的话没有）。若有参数，则将参数写入到data中，调用transact方法发起RPC(远程过程调用)请求，同时当前的线程挂起；服务端的onTransact方法会被调用，直到RPC过程返回后，当前线程继续执行，并从reply中取出返回结果，然后返回数据

总结

工作流程图如下：

在这个Binder实例中，Stub是真正继承Binder的类，与Binder是is关系；而Proxy只是持有一个Binder对象，它拥有Binder的能力，但本身正如其名字一样，只是一个代理，与Binder关系是has。在了解Binder的设计理念后可以轻松理解这样设置的意义，在Client端中我们只需要Binder的能力，因为在这里只是记录操作，真正的操作要到服务端中执行，所以服务端需要一个Binder对象。这也就是最开始那句话的解释

AIDL只是一种辅助手段，我们完全可以自己实现Binder，只不过写出来会发现和IDE生成的一模一样(除了命名)，手写的意义在于加深理解。
